## CRANE の三層メモリは「速度のため」ではなく「文脈のため」に三つに割っている

> ※ 本文は CRANE における三層メモリ設計の説明を引用したものである。
> CYRUNE はこの三層契約を利用するが、CRANE Kernel はあくまで契約（trait）のみを定義し、本文中の具体実装例（RocksDB / BatchEngine / 数値目標等）は reference implementation の説明であり、Kernelの固定仕様ではない。

CRANE が解こうとしている問題は単純で、**LLM/人間の作業文脈は“全部は持てない”**という制約です。だから、メモリを一つの巨大ストアとして扱うのではなく、人間の記憶モデルに合わせて、役割・寿命・容量・読み書きコストが違う三つに分けます。三層は、単なるキャッシュ階層ではなく、**「何を今の判断材料として扱うか」**を物理的に固定するための設計です。全体像はこのように定義されています：Working は 10±2 の作業記憶、Processing は 42 日の中期保持、Permanent は長期保存（永続・圧縮・忘却を含む）です。

この分割は「高速化のための都合」ではなく、**“混ぜると破綻する責務”を分離**するための都合です。たとえば、永続化・圧縮・索引更新・バッチ処理といった重い処理を、ユーザーが待っているホットパス（対話・推論の即応）に混ぜると、P95/P99 が跳ねて体験が壊れます。そこで CRANE は、ホットパスと重処理を分け、重処理はバッチ化して後段に逃がす、という原則を明記しています。

---

## 1) Working Memory：10±2 を「絶対に」超えない、今この瞬間の文脈

Working Memory は、**“今まさに必要な情報だけ”**を保持する作業記憶です。容量は Miller’s Law に基づき、**8〜12（代表 10±2）**の範囲を守るものとして説明されます。

ここで重要なのは、Working を「小さなキャッシュ」と誤認すると設計も検証も全部壊れる点です。CRANE は明確に、**Working に 1000 件挿入してヒット率を見るのは誤り**で、そういう評価は Processing/Permanent の仕事だ、と釘を刺しています。
つまり Working は「多く当てる」ためではなく、**“混ぜない（ノイズを増やさない）”ために小さく保つ**層です。

実装的には、超短経路（L1 cache / lock-free / SIMD / zero-copy 的な扱い）で **<200ns 級の参照**を目標にする、といった記述が並びます。
ただし、ホットパスで時刻取得や重い統計や複雑な置換ロジックをやり始めるとそれ自体がボトルネックになるため、**観測（metrics）と性能の両立**として「重い集計は別スレッド/バッチに逃がす」ことが共通原則として書かれています。

---

## 2) Processing Memory：42 日保持の「中期」、重処理をバッチで捌く中核

Processing Memory は、Working に載せっぱなしにすると作業文脈が汚れるが、捨てるには惜しい情報（推論途中の成果、セッション内の処理結果、しばらく参照される履歴）を保持する層です。定義上、**42 日保持**が中核要件として扱われます。

この 42 日は「なんとなく」ではなく、忘却モデル（Ebbinghaus）に AI 緩和係数 α=0.25 を入れて保持を伸ばす、という建付けで説明され、実装例でも「係数 0.25」「42 日で確実に削除」がそのまま登場します。
要するに Processing は **“永遠ではないが長めに残す”**ための層で、ここに置く情報は「将来の参照可能性」と「作業文脈の純度」のバランスで決まります。

Processing の設計が面白いのは、性能の出し方が「ホットパス最適化」ではなく、**バッチ処理でコストをならす**方向に固定されている点です。Processing では、永続化/圧縮/DB 書き込みなどの重い I/O を **Batch 化して高効率にさばく**ことが責務とされ、BatchProcessingEngine（OperationQueue→Coordinator→WriteBatchOptimizer→FlushTrigger…）の分解設計が明記されています。
実際、バッチ処理の性能目標として **3,376,000 ops/sec**、P99 < 0.2ms といった数字が提示されています。

そして、Processing には「実装の落とし穴」がはっきり存在します。DirectMode（即時書き込み）と BatchedMode（バッチ）を切り替える設計がある一方、**BatchedMode で保存カウントは増えるのに取得が 0 件＝保持精度 0%**という致命障害が記録されており、検証結果として Direct=100% / Batched=0% が明記されています。
この手の不具合は「速い・遅い」以前に **“永続化された事実があるか”**の検証を要求します（RocksDB の Column Family を直接読んで物理データ有無を確認する手順まで書かれている）。

## 3) Permanent Memory：長期保存。遅くてよい、ただし消えない／探せる

Permanent Memory は、長期保存（知識ベース、永続的データ）を担います。ここは「低頻度アクセスでよい」代わりに、**永続化・圧縮・忘却・索引**を含むライフサイクル管理が責務です。

ストレージとして RocksDB が例示され、圧縮（LZ4/Snappy）や、Ebbinghaus 忘却（AI 緩和 0.25）が統合される、と説明されます。
また、Permanent の性能については「<5ms 目標が厳しすぎる可能性があり、Permanent は <10ms 程度へ再設定する提案がある」という設計判断も残っています（＝Permanent は Working と同じ基準で縛らない）。

---

## 三層を「つなぐ」ルール：昇格・降格・忘却は数式と閾値で決める

三層で最も重要なのは「個別の実装」よりも **層間移動**です。CRANE には、Working→Processing、Processing→Permanent の移行閾値や、予測プリロード、重複排除などをまとめた設定例があり、たとえば **Working→Processing を 0.8、Processing→Permanent を 0.6**のように閾値で書かれています。
さらに統合アクセスの擬似コードでは、検索順序は Working→Processing→Permanent で、Processing で当たったものを **条件により Working にプロモートする**流れが示されています。

忘却は「削除」ではなく、保持・昇格・降格の判断軸です。Processing の例では「42 日で確実に削除」「それ以外は retention を計算して重要度と合わせて判定」という形で書かれており、Ebbinghaus + AI 緩和係数 0.25 が具体的に使われています。

---

## 三層を成立させるための“共通フォーマット”：ID とメタデータの一貫性

三層は別々の器ですが、**同じ“物（MemoryItem）”が移動する**ので、共通のデータ形が必要です。CRANE の仕様では MemoryId は **u64**で、**0 は予約値として回避**する、といった具体がデータフォーマットとして書かれています。
同様に MemoryType は Working/Processing/Permanent の閉集合で表現されます。

（補足）あなたの現行方針としては「コンテキストウィンドウは毎ターン クリアし、保持すべき状態は Working Memory 側へ移す」が前提なので、実運用上は Working が「ターンを跨ぐ唯一の揮発性の受け皿」になります。その上で、Processing/Permanent は“より長い寿命の貯蔵・検索”として働く、という構図になります。

## 典型シナリオで理解する（実際に何がどこへ入るか）

例：あなたがエージェントに「新しい機能の設計レビュー」を依頼する。

1. 直近の要求、制約、未決論点、今出ている候補案だけを Working に置く（10±2 を超えない）。
2. レビュー途中で出た比較表、判断理由、検討ログ、次に参照しそうなリンク類は Processing に落ちる。ここは 42 日生きるので「1か月後に再検討」でも残る。
3. 設計が確定した ADR 相当、外部に説明可能な結論、将来の監査対象になる決定は Permanent に入る（圧縮・索引・忘却の対象になりつつ基本は残る）。

このとき、あなたが「Working に 1000 件入れて検索精度を上げよう」と考え始めたら、それは CRANE の前提に反していて、設計もベンチも誤ります（Working の誤用として明記されています）。

---

## 最後に：三層メモリの“設計の勝ち筋”は、誤用を物理的に起こせないようにすること

三層メモリの価値は「速い」より先に、**どこに何を置くべきかが強制される**ことです。

* Working を小さく保てば、判断が速くなり、ノイズ混入が減る（10±2 を守る）。
* Processing で重処理をバッチ化すれば、ホットパスの尾部レイテンシが守られる（BatchEngine で捌く）。
* Permanent に「消えないもの」を集めれば、時間が経っても再利用できる（永続・圧縮・忘却）。
* そして忘却は、雑に delete するのではなく、42 日・重要度・アクセスで“保持”を制御する（Ebbinghaus + α=0.25）。
